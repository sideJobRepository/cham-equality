# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & run

- Java 25 (Gradle toolchain) / Spring Boot 4.0.3 / Gradle wrapper
- Run app: `./gradlew bootRun` (default profile reads `application.yml`; activate `real` with `--args='--spring.profiles.active=real'`)
- Build: `./gradlew build`
- Test: `./gradlew test` — single test: `./gradlew test --tests 'com.chamapi.file.service.FileBatchServiceTest.test1'`
- QueryDSL Q-classes: generated by `annotationProcessor` (`io.github.openfeign.querydsl:querydsl-apt:6.11:jpa`) during `compileJava`. Delete `build/generated` if Q-classes go stale.

## Configuration

- `.env` (loaded via `spring-dotenv`) supplies `DB_URL`, `DB_PORT`, `DB_USERNAME`, `DB_PASSWORD`, `AWS_ACCESS_KEY`, `AWS_SECRET_KEY`, `AWS_REGION`, `AWS_BUCKET`, `COOKIE_SECURE`.
- `ddl-auto: none` — schema is managed manually (see `README-db.md` for DB conventions: BIGINT PKs, `VARCHAR(500)` default, `CREATE_DATE`/`MODIFY_DATE` on every table, SNAKE_CASE uppercase tables/columns).
- Tests use `@ActiveProfiles("test")` against `src/test/resources/application-test.yml`, which still hits the real DB via `.env` (no H2/testcontainers).
- JPA SQL logs go through `log4jdbc` (`DriverSpy`) — see `src/main/resources/log4jdbc.log4j2.properties` and `common/log/CustomLog4JdbcCustomFormatter.java`.

## Package layout

Top-level packages under `com.chamapi` are **feature modules** (`file`, `member`, `authentication`, `authorization`, `security`, `shelter`, `common`). Each module uses the layered split `controller` / `service` (+ `impl`) / `repository` / `entity` / `dto` / `config` / `enums`.

- **`common`**: `DateSuperClass` (`@MappedSuperclass` with `@CreatedDate`/`@LastModifiedDate` — every entity extends it), global `ExceptionController` (`@RestControllerAdvice` mapping `CustomException` subclasses to `ErrorMessageResponse`), `CacheConfig` (Caffeine), and the AOP JPA query-context interceptor in `common/log`.
- **`security`** vs **`authentication`** vs **`authorization`** are separate — don't merge them:
  - `security/` = Spring Security wiring (filters, providers, handlers, JWT signer/decoder, social login services).
  - `authentication/` = refresh-token domain (entity, service, cookie factory via `AuthProperties`).
  - `authorization/` = URL↔Role mapping domain (`UrlResources`, `UrlResourcesRole`, `Role`, `RoleHierarchy`, `MemberRole` entities).

## Architectural notes that span files

### Dynamic URL-based authorization
`security.SecurityConfig` wires `.anyRequest().access(chamEqualityAuthorizationManager)`. `ChamEqualityAuthorizationManager` loads `Map<String,String>` of `"<METHOD> <path>" → "ROLE_X"` (or SpEL expression) from the DB via `DynamicAuthorizationServiceImpl` in `@PostConstruct`/`reload()`, then builds `RequestMatcherEntry`s using `PathPatternRequestMatcher`. To change URL→role rules at runtime, update the DB tables and call `reload()`; don't add `authorizeHttpRequests` matchers in `SecurityConfig`.

### JWT signing with RSA
Public key is a file (`src/main/resources/keys/public.pem`); private key is stored **in the DB** and loaded via `RsaRepository.findAll().get(0).getRsaPrivateKey()` inside `SignatureConfig#rsaKey()`. `JwtAuthenticationConverter` reads authorities from the `roles` claim with **no prefix** (not `SCOPE_`/`ROLE_`). Access tokens are issued by `RsaSecuritySigner`; refresh tokens live in DB (`authentication.RefreshToken`) and in a cookie configured by `AuthProperties` (`auth.cookie.*`, `auth.refresh-token.expiry`).

### Custom auth flow
`SecurityDsl` registers `ChamEqualityAuthenticationFilter` for login. `SocialAuthenticationProvider` delegates to `KaKaoServiceImpl`/`NaverServiceImpl` under `security/service/impl/` (interfaces in `security/service/social/`). On success, `chamEqualityAuthenticationSuccessHandler` issues the JWT pair; failure/entrypoint/access-denied are all custom beans wired in `SecurityConfig`.

### QueryDSL repository pattern
Each JPA repo pairs three files, e.g. for `CommonFile`:
- `repository/CommonFileRepository.java` — `extends JpaRepository<CommonFile, Long>, CommonFileQueryRepository`
- `repository/query/CommonFileQueryRepository.java` — interface with custom methods
- `repository/impl/CommonFileRepositoryImpl.java` — uses `JPAQueryFactory` (bean defined in `ChamApiApplication`)

Spring Data auto-wires the `*Impl` by naming convention. Follow this trio for any new custom queries; don't put QueryDSL code directly in the `JpaRepository` interface.

### S3 presigned-URL caching
`FileViewUrlCache#getFileUrl(Long)` is `@Cacheable(cacheNames = CacheConfig.FILE_VIEW_URL, key = "#id")`. Cache TTL (`FILE_VIEW_URL_TTL = 50m`) is passed as the presigned-URL `signatureDuration`, so cache entry and S3 signature expire together. `CommonFileSchedule` runs daily at 01:00 Asia/Seoul to purge `TEMPORARY` files older than a day via `FileBatchService.temporaryFileRemove`.

## Conventions

- Entities extend `DateSuperClass` for audit dates; column names use SNAKE_CASE uppercase (`SHELTER_ID`, `CREATE_DATE`).
- Throw subclasses of `common.exception.CustomException` (override `getStatus()`); `ExceptionController` turns them into `ErrorMessageResponse`. `ValidationException` is available for field-level validation errors.
- Services are `@Transactional(readOnly = true)` at class level with `@Transactional` on write methods — follow this in `RefreshTokenServiceImpl` style.
- `common.dto.ApiResponse` is the standard envelope; `ErrorMessageResponse` is the error envelope with a `validation` field list.
- Test base classes live at `src/test/java/com/chamapi/` (e.g. `RepositoryAndServiceTestSupport`, `ControllerTestSupport`) — extend them instead of re-annotating `@SpringBootTest` each time.
